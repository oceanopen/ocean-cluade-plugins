实践一

编写高质量 Prompt

✓ 推荐做法

  • 提供具体的文件路径和行号
  • 描述预期行为而非实现细节
  • 给出错误日志或堆栈信息
  • 说明已尝试过的方案
  • 指定技术栈和版本约束
  • 拆分复杂任务为多个步骤

✗ 常见误区

  • 模糊描述："帮我改一下那个功能"
  • 缺少上下文："这个 bug 怎么修"
  • 一次要求完成 10 个功能
  • 不提供错误信息就问"为什么不工作"
  • 过度限制实现方式
  • 忽略项目已有的模式和约定
💡
核心原则：把 Claude Code 当作一位刚加入团队的资深工程师 — 他能看到代码，但不知道会议讨论了什么、为什么这样设计。给他足够的 上下文 和 意图。

实践二

用 CLAUDE.md 建立项目共识

📋 CLAUDE.md 推荐结构

# 项目概述
电商平台后端，Node.js + TypeScript

# 技术栈
- Runtime: Node 20+
- Framework: Fastify 4.x
- ORM: Prisma 5.x
- Testing: Vitest

# 代码规范
- 使用 ESM 模块格式
- 错误处理用 Result 模式
- API 路由遵循 RESTful 约定

# 常用命令
- pnpm dev: 启动开发服务器
- pnpm test: 运行测试
- pnpm db:migrate: 数据库迁移

🎯 关键配置层级

  • G
    ~/.claude/CLAUDE.md
    全局指令，所有项目生效（个人偏好）
  • P
    项目根目录/CLAUDE.md
    项目级约定（提交到 Git，团队共享）
  • D
    子目录/CLAUDE.md
    模块级指令（按需添加）
  • U
    用户提示中
    单次对话的临时指令
🔑
最佳实践：把 CLAUDE.md 当作给新成员的 README++，包含项目约定、常见命令和注意事项。

实践三

高效上下文管理

📦

自动压缩

对话过长时，Claude Code 会自动压缩早期内容，保留关键摘要。你无需手动管理，但要注意关键文件可能被总结。

🎯

精准引用

指定文件路径和行号，而非让 Claude 搜索整个项目。减少不必要的上下文消耗，提高响应精度。

🔄

适时重启

切换到完全不同的任务时，开启新对话而非延续旧会话。CLAUDE.md 会自动加载，不会丢失项目上下文。

上下文管理策略一览

策略 场景 效果
CLAUDE.md 项目约定、技术栈说明 ✓ 每次对话自动加载
/compact 命令 对话过长、响应变慢时 ✓ 手动触发压缩
指定文件路径 需要精确操作时 ✓ 减少搜索开销
Subagent 分派 并行探索多个方向 ✓ 隔离上下文消耗
Memory 系统 跨会话持久化信息 ✓ 长期知识积累

实践四

权限与安全管理

🛡️ 权限模式

Default 模式 推荐

文件读取自动允许，文件写入和命令执行需要确认。日常开发的最佳平衡。

Auto-accept 模式 进阶

自动接受文件编辑。适合对项目非常熟悉、信任 Claude 输出的场景。

Plan 模式 规划

只读取和分析，不做任何修改。适合代码审查、架构分析等只读场景。

⚙️ settings.json 配置

// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(pnpm test)",
      "Bash(pnpm lint)",
      "Bash(git status)",
      "Bash(git diff)",
      "Bash(git log*)"
    ],
    "deny": [
      "Bash(rm -rf*)",
      "Bash(drop*)",
      "Bash(*--force*)"
    ]
  }
}
⚠️
安全红线：绝不允许 Claude 直接操作生产环境数据库、执行 rm -rf 或 force push。用 deny 列表防护。

实践六

善用 Memory 系统

🧠 四类记忆

👤
User Memory

用户角色、偏好、专业领域。
例："我是 Go 后端工程师，对 React 比较陌生"

📢
Feedback Memory

用户对输出方式的反馈。
例："不要在每次回复后加总结"

📋
Project Memory

项目状态和上下文。
例："API v2 迁移计划截止 6 月 15 日"

🔗
Reference Memory

外部资源指针。
例："Pipeline bug 追踪在 Linear INGEST 项目"

✨ 最佳实践

  • 1
    主动告诉 Claude "记住这个"，重要信息会自动持久化
  • 2
    纠错和确认都值得记录 — 不只记错误，也记成功的判断
  • 3
    定期清理过时的记忆，保持信息准确
  • 4
    用 /memory 查看当前记忆状态
💡
Memory 适合存储非代码信息。代码约定和项目结构应该放在 CLAUDE.md，而非 Memory。

实践七

Hooks 实现工作流自动化

⚙️ Hooks 配置示例

// .claude/settings.json
{
  "hooks": {
    "afterWrite": [
      {
        "matcher": "*.ts",
        "command": "pnpm lint $FILE"
      }
    ],
    "afterBash": [
      {
        "matcher": "pytest*",
        "command": "echo 'Tests done'"
      }
    ]
  }
}

🎯 常用 Hook 场景

  • 1
    写文件后自动格式化
    保存 .ts 文件后自动运行 prettier
  • 2
    提交前运行测试
    Git 操作前自动跑测试套件
  • 3
    安全检查
    编辑敏感文件时弹出确认
  • 4
    自定义通知
    长时间任务完成后推送通知

避坑指南

常见 反模式与对策

🚫 反模式

  • ❌ 盲目信任输出 — 不 review 就提交
  • ❌ 一句话需求 — "做一个登录页" 没有任何细节
  • ❌ 忽略安全审查 — 跳过 security review
  • ❌ 过度依赖 — 连一行 CSS 都让 AI 写
  • ❌ 不设权限 — Auto-accept 一切操作
  • ❌ 泄露敏感信息 — 在 prompt 中粘贴密钥

✅ 正确做法

  • ✓ Review 每次变更 — 养成检查 diff 的习惯
  • ✓ 结构化需求 — 背景 + 目标 + 约束
  • ✓ 定期安全审计 — 用 /security-review
  • ✓ AI 辅助 ≠ AI 代替 — 保持判断力
  • ✓ 合理权限 — deny 危险操作
  • ✓ 使用 .gitignore — 排除敏感文件
🎯
核心原则：把 Claude Code 看作一个能力很强但需要指导的协作者，而不是一个不需要监督的自动编码机。你是架构师，它是实现者。
