# Frontend Slides — HTML 演示文稿生成器

创建零依赖、动画丰富的 HTML 演示文稿，完全在浏览器中运行。

> 来源：[frontend-slides](https://github.com/zarazhangrui/frontend-slides)

## 核心原则

1. **零依赖** — 单个 HTML 文件，内联 CSS/JS，无需 npm 或构建工具。
2. **展示而非描述** — 生成可视化预览，而非抽象选择。人们通过看到实物来发现自己的喜好。
3. **独特设计** — 拒绝千篇一律的"AI 风格"，每份演示文稿都应是定制化作品。
4. **渐进式披露** — 先读取轻量级样式索引，用户选择后再加载完整的设计文档。
5. **固定 16:9 舞台（不可协商）** — 所有演示文稿使用 1920×1080 幻灯片画布，整体缩放适配视口。幻灯片在所有设备上必须保持 16:9 比例，不得根据设备重新排版内容。

## 设计美学

避免千篇一律的 AI 生成风格，追求创意独特、令人惊喜的设计。

**重点方向**：

- **字体**：选择独特、美观的字体。避免 Arial、Inter 等通用字体，选择能提升美感的独特选项。
- **配色与主题**：坚持统一的美学风格，使用 CSS 变量保持一致性。主导色 + 锐利强调色优于均匀分布的柔和调色板。
- **动效**：使用动画增加效果和微交互。优先使用纯 CSS 方案。聚焦高影响力时刻：一个精心编排的页面加载 + 交错显示比零散的微交互更能带来愉悦感。
- **背景**：营造氛围和深度，而非默认纯色。叠加 CSS 渐变、使用几何图案或添加与整体美学匹配的上下文效果。

**应避免的通用 AI 美学**：

- 过度使用的字体系族（Inter、Roboto、Arial、系统字体）
- 陈词滥调的配色方案（尤其是白色背景上的紫色渐变）
- 可预测的布局和组件模式
- 缺乏上下文特色的模板化设计

## 固定舞台规则

以下不变量适用于每份演示文稿的每张幻灯片：

- 每份演示文稿都有一个填满浏览器窗口的视口包装器
- 每张幻灯片在固定的 1920×1080 舞台内创作
- 舞台均匀缩放以适配视口，可以留黑边，但不能重新排版内容
- 不要使用响应式断点来重新排列手机上的幻灯片内容
- 在 1920×1080 设计尺寸下使用固定的内部幻灯片尺寸
- 幻灯片可见性通过 `.active` / `.visible` 使用 `visibility`、`opacity` 和 `pointer-events` 控制，来自 `viewport-base.css`。不要使用 `display: none` / `display: block` 切换幻灯片
- `clamp()` 仅用于舞台外的非幻灯片 UI
- 包含 `prefers-reduced-motion` 支持
- 永远不要直接取反 CSS 函数（`-clamp()`、`-min()`、`-max()` 会被静默忽略），使用 `calc(-1 * clamp(...))` 代替

**生成时，读取 `viewport-base.css` 并将其完整内容包含在每份演示文稿中。**

### 内容密度模式

询问用户这是主要用来阅读还是用来演讲，然后据此设计：

| 密度模式 | 适用场景 | 设计行为 |
| -------- | -------- | -------- |
| **低密度 / 演讲型** | 公开演讲、Keynote 分享、现场讲解 | 每页一个观点、大号字体、强视觉层次、充足留白、最多 1-3 个要点 |
| **高密度 / 阅读型** | 报告、传阅材料、异步审阅、详细内部文档 | 更自包含的幻灯片、结构化网格/表格/注释、4-8 个要点或 4-6 张卡片 |

基准限制仍然适用：不允许滚动、溢出、面板重叠，文字不得低于舒适阅读大小。内容超出所选密度模式时，拆分为更多幻灯片。

## 阶段 0：检测模式

确定用户需求：

- **模式 A：新建演示文稿** — 从零创建。进入阶段 1。
- **模式 B：PPT 转换** — 转换 .pptx 文件。进入阶段 4。
- **模式 C：增强改进** — 改进现有 HTML 演示文稿。读取、理解后增强。**遵循模式 C 修改规则。**

### 模式 C：修改规则

增强现有演示文稿时，固定舞台适配是最大风险：

1. **添加内容前**：统计现有元素，检查密度限制
2. **添加图片**：适配 1920×1080 幻灯片画布。如已满则拆分为两张
3. **添加文字**：每张最多 4-6 个要点。超限则拆分
4. **任何修改后验证**：幻灯片舞台保持 16:9、无文字溢出、无面板重叠
5. **主动重组**：修改可能导致溢出时，自动拆分内容并告知用户

## 阶段 1：内容发现（新建演示文稿）

**一次性提出所有问题**：

**问题 1 — 用途**：演示文稿的用途是什么？选项：路演 / 教学教程 / 会议演讲 / 内部分享

**问题 2 — 篇幅**：大约多少页？选项：短篇 5-10 / 中篇 10-20 / 长篇 20+

**问题 3 — 内容**：内容准备好了吗？选项：全部就绪 / 粗略笔记 / 仅有主题

**问题 4 — 密度**：演示文稿的密度？选项：
- "低密度 / 演讲型" — 大想法、少文字、更多视觉呼吸空间
- "高密度 / 阅读型" — 更自包含的细节，适合异步阅读

**不要在阶段 1 询问行内编辑。** 行内编辑是草稿后的功能：默认包含，除非用户明确要求锁定/仅导出文件。

记住用户的密度选择，它影响幻灯片数量、字体大小、每页文字量和布局密度。

### 步骤 1.2：图片评估（如果提供了图片）

如果用户未提供图片 → 跳到阶段 2。

如果用户提供图片文件夹：

1. **扫描** — 列出所有图片文件
2. **检查每张图片** — 使用图片理解能力分析内容
3. **评估** — 每张图片：展示什么、可用/不可用、代表什么概念、主色调
4. **协同设计大纲** — 图片和文字共同决定幻灯片结构
5. **确认大纲** — "这个幻灯片大纲和图片选择看起来如何？"

---

## 阶段 2：风格发现

**这是"展示而非描述"阶段。** 大多数人无法用语言表达设计偏好。

### 步骤 2.0：直接生成 3 个风格预览

根据用途、受众、情绪和内容密度，生成 3 个不同的单页 HTML 预览，展示字体、配色、动画和整体美学。

不要问用户是否想要选项或预设选择器。默认的发现体验始终是视觉对比。

如果用户已给出风格感觉，使用它。否则从场合、受众、内容推断可能的情绪。

读取 [STYLE_PRESETS.md](STYLE_PRESETS.md) 获取安全预设候选。如果 [bold-template-pack/selection-index.json](bold-template-pack/selection-index.json) 存在，也读取该紧凑索引，但暂不读取任何 `design.md` 文件。

| 情绪 | 推荐预设 |
| ---- | -------- |
| 印象深刻/自信 | Bold Signal、Electric Studio、Dark Botanical |
| 兴奋/活力 | Creative Voltage、Neon Cyber、Split Pastel |
| 平静/专注 | Notebook Tabs、Paper & Ink、Swiss Modern |
| 受启发/感动 | Dark Botanical、Vintage Editorial、Pastel Geometry |

**预览混合规则**：

- 默认生成 3 个预览：1 个来自 `STYLE_PRESETS.md` 的安全预设，至少 1 个来自 `bold-template-pack/selection-index.json` 的大胆模板，1 个自由发挥
- 三个预览之间保持真正的差异化

**预览真实性规则（不可协商）**：

- 每个风格预览必须像用户演示文稿中真实的首页，而非诊断卡片
- 永远不要在幻灯片上渲染内部工作流文本（如 preview、Option A/B/C、文件名等）
- 如果幻灯片需要装饰元素，只使用真实的演示文稿装饰：标题、日期、作者、页码等

将预览保存到 `.frontend-slides/slide-previews/`（style-a.html、style-b.html、style-c.html），每个都是自包含的紧凑单页标题幻灯片。

自动为用户打开每个预览。

### 步骤 2.1：用户选择

询问用户偏好哪个风格预览。如果选择"混合元素"，询问具体细节。

---

## 阶段 3：生成演示文稿

使用阶段 1 的内容和阶段 2 的风格生成完整演示文稿。

贯穿整个演示文稿应用用户的密度选择：

- **低密度 / 演讲型**：使用更多幻灯片，每页更少想法。大标题、短语、视觉隐喻、引用页、演讲友好的节奏。
- **高密度 / 阅读型**：幻灯片更自包含。结构化网格、对比表格、标注图表、简洁说明文字。

如果用户选择了 `bold-template-pack` 中的大胆模板，生成前读取该模板的完整 `design.md`。将 `design.md` 视为设计配方：

- 保留其字体、调色板、装饰词汇、间距节奏和组件语法
- 生成固定 1920×1080 舞台、均匀缩放适配视口的最终演示文稿
- 将 `design.md` 中的视口流式值视为设计比例，转换为 1920×1080 舞台坐标
- 保持输出为单个自包含的 HTML 文件
- 生成后验证内容溢出和面板重叠

**生成前，读取以下支持文件**：

- [html-template.md](html-template.md) — HTML 架构和 JS 功能
- [viewport-base.css](viewport-base.css) — 必须包含的 CSS（完整包含）
- [animation-patterns.md](animation-patterns.md) — 动画参考

**关键要求**：

- 单个自包含 HTML 文件，所有 CSS/JS 内联
- 在 `<style>` 块中包含 viewport-base.css 的完整内容
- 使用 Fontshare 或 Google Fonts 字体 — 不使用系统字体
- 每个部分添加清晰的 `/* === SECTION NAME === */` 注释块

---

## 阶段 4：PPT 转换

转换 PowerPoint 文件时：

1. **提取内容** — 运行 `python scripts/extract-pptx.py <input.pptx> <output_dir>`（如需要安装 python-pptx：`pip install python-pptx`）
2. **与用户确认** — 展示提取的幻灯片标题、内容摘要和图片数量
3. **风格选择** — 进入阶段 2 进行风格发现
4. **生成 HTML** — 转换为选定风格，保留所有文字、图片、幻灯片顺序和演讲者备注

---

## 阶段 5：交付

1. **清理** — 删除 `.frontend-slides/slide-previews/`（如存在）
2. **打开** — 使用 `open [filename].html` 在浏览器中启动
3. **总结** — 告知用户：
   - 文件位置、风格名称、幻灯片数量
   - 导航方式：方向键、空格键、滑动/点击
   - 自定义方式：`:root` CSS 变量改颜色、字体链接改排版、`.reveal` 类改动画
   - 行内文本编辑可用：悬停左上角或按 E 进入编辑模式，点击文字即可编辑，Ctrl+S 保存

---

## 阶段 6：分享与导出（可选）

交付后询问用户是否需要分享或导出。

- **部署到 URL** — 可分享链接，在任何设备上可用
- **导出为 PDF** — 通用文件，适用于邮件、Slack、打印
- **两者都要**
- **不需要**
